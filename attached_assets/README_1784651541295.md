## 1. Instrução principal para a IA

Crie um portfólio profissional, moderno, responsivo e acessível para **Pedro, Desenvolvedor Full Stack com foco em interfaces web e desenvolvimento de APIs**.

O portfólio será utilizado principalmente para conquistar clientes em plataformas de freelancer. A aplicação deve transmitir confiança, organização, responsabilidade, comprometimento e atenção aos detalhes.

O objetivo não é apenas apresentar tecnologias. O portfólio deve ajudar o visitante a compreender rapidamente:

- quem é o profissional;
- quais problemas ele pode resolver;
- quais serviços oferece;
- como desenvolve seus projetos;
- por que é confiável;
- como entrar em contato.

A principal conversão esperada é:

> Fazer o visitante iniciar uma conversa sobre um projeto ou solicitar um orçamento.

---

## 2. Regra de fidelidade das informações

A IA não deve inventar:

- experiências profissionais;
- clientes;
- empresas;
- depoimentos;
- métricas;
- resultados;
- anos de experiência;
- certificações;
- projetos;
- habilidades;
- links;
- informações pessoais;
- nível de domínio técnico.

Quando uma informação real não estiver disponível, utilizar um placeholder explícito:

```text
[SEU_NOME_COMPLETO]
[SEU_EMAIL]
[SEU_WHATSAPP]
[URL_DO_GITHUB]
[URL_DO_LINKEDIN]
[URL_DO_CURRÍCULO]
[URL_DO_PROJETO]
[URL_DO_REPOSITÓRIO]
[DESCRIÇÃO_REAL_DO_PROJETO]
[TECNOLOGIAS_DO_PROJETO]
```

Nunca publicar placeholders como se fossem informações definitivas.

---

## 3. Objetivo do produto

O visitante deve conseguir entender, preferencialmente nos primeiros segundos:

1. quem é Pedro;
2. que tipo de trabalho ele realiza;
3. quais soluções pode desenvolver;
4. onde visualizar os projetos;
5. como entrar em contato.

O portfólio deve conduzir o visitante naturalmente da apresentação até o contato.

---

## 4. Público-alvo

O portfólio deve atender:

- clientes de plataformas freelancer;
- pequenos negócios;
- empresas;
- startups;
- profissionais liberais;
- empreendedores;
- pessoas que precisam de soluções web personalizadas.

Não limitar o posicionamento a um nicho específico.

---

## 5. Posicionamento profissional

Utilizar como título profissional:

> Desenvolvedor Full Stack

Complementar o posicionamento com:

> Com foco em interfaces modernas e desenvolvimento de APIs.

Não utilizar:

- especialista Full Stack;
- especialista em sistemas completos;
- domínio avançado de todas as camadas;
- especialista em integração Front-end e Back-end;
- soluções completas de ponta a ponta;
- qualquer afirmação que exagere a experiência atual.

Não apresentar Rust como tecnologia dominada profissionalmente. Rust deve aparecer exclusivamente como uma tecnologia em estudo.

---

## 6. Personalidade da marca

A identidade deve transmitir:

- confiança;
- responsabilidade;
- comprometimento;
- organização;
- clareza;
- profissionalismo;
- modernidade;
- evolução constante.

A experiência deve ser elegante e discreta.

Não criar uma aparência:

- gamer;
- cyberpunk;
- neon;
- excessivamente futurista;
- exageradamente corporativa;
- visualmente poluída;
- baseada em efeitos chamativos;
- semelhante a um template genérico de portfólio.

O conceito central do projeto é:

> Cada detalhe comunica confiança.

---

## 7. Princípios de decisão

Quando houver mais de uma solução possível, seguir esta ordem de prioridade:

1. Correção
2. Acessibilidade
3. Clareza
4. Usabilidade
5. Performance
6. Manutenibilidade
7. Consistência
8. Estética
9. Animação

Regras obrigatórias:

- priorizar soluções simples;
- não adicionar funcionalidades fora do escopo;
- não aumentar a complexidade para demonstrar conhecimento técnico;
- não utilizar bibliotecas sem necessidade real;
- não esconder conteúdo importante atrás de animações;
- não sacrificar legibilidade por estética;
- não sacrificar acessibilidade por animação;
- não criar abstrações prematuras;
- evitar padrões arquiteturais desnecessários;
- escolher a opção de menor custo de manutenção quando as alternativas forem equivalentes.

---

## 8. Stack tecnológica

Utilizar:

```text
React
TypeScript
Vite
Tailwind CSS
Lucide React
```

Utilizar preferencialmente:

- CSS para transições simples;
- Intersection Observer para animações de entrada;
- APIs nativas do navegador quando forem suficientes.

Framer Motion não é uma dependência obrigatória.

Adicionar Framer Motion somente quando:

- houver uma necessidade concreta;
- simplificar significativamente uma animação;
- não prejudicar performance;
- não aumentar desnecessariamente o bundle;
- respeitar `prefers-reduced-motion`.

Não utilizar:

- Bootstrap;
- jQuery;
- bibliotecas de componentes completas sem necessidade;
- dependências apenas para funcionalidades simples;
- frameworks de animação quando CSS resolver adequadamente.

---

## 9. Estrutura da página

A aplicação deve ser uma landing page de portfólio com as seguintes seções:

1. Header
2. Hero
3. Sobre mim
4. Como posso ajudar você?
5. Projetos
6. Tecnologias
7. Contato
8. Footer

A seção de projetos deve receber maior destaque visual do que a seção de tecnologias.

---

## 10. Header

O Header deve conter:

- identificação ou logotipo textual;
- links para as principais seções;
- CTA de contato;
- menu mobile acessível.

Links recomendados:

```text
Sobre
Serviços
Projetos
Tecnologias
Contato
```

Requisitos:

- navegação com âncoras;
- rolagem suave somente quando não houver redução de movimento;
- indicação clara de foco;
- menu mobile navegável por teclado;
- fechamento pelo botão, pela tecla `Escape` e após a seleção de um link;
- bloqueio apropriado da rolagem quando o menu mobile estiver aberto;
- não depender apenas de ícones sem rótulos acessíveis.

O Header pode se tornar fixo ou semitransparente durante a rolagem, desde que mantenha boa legibilidade e não utilize glassmorphism exagerado.

---

## 11. Hero

A Hero deve apresentar uma narrativa curta e direta.

### Conteúdo recomendado

Pergunta principal:

> O que sua ideia precisa para se tornar realidade?

Texto com efeito de digitação:

```text
Uma Landing Page.
Um Sistema Web.
Uma API.
Um Dashboard.
Uma solução moderna.
```

Após ou junto ao efeito, apresentar:

> Olá, eu sou Pedro.

Título:

> Desenvolvedor Full Stack

Descrição:

> Desenvolvo interfaces modernas, APIs e aplicações web com foco em qualidade, código organizado e uma experiência que inspira confiança.

CTAs:

```text
Ver projetos
Vamos conversar
```

### Regras da Hero

- o nome e o posicionamento devem permanecer visíveis;
- o visitante não deve depender do efeito de digitação para entender o conteúdo;
- os botões devem estar disponíveis imediatamente;
- não aguardar o término da animação para revelar informações essenciais;
- utilizar no máximo cinco frases no efeito;
- não usar tecnologias no efeito de digitação;
- não forçar `height: 100vh`;
- utilizar uma altura mínima adequada ao conteúdo;
- considerar a área útil do navegador mobile;
- evitar ilustrações genéricas;
- não utilizar uma fotografia sem que ela seja fornecida;
- não criar avatar fictício para representar Pedro.

Quando `prefers-reduced-motion: reduce` estiver ativo:

- remover o efeito de digitação;
- exibir uma frase estática;
- remover animações não essenciais.

---

## 12. Sobre mim

A seção deve reunir apresentação, trajetória e diferenciais sem criar seções adicionais desnecessárias.

Título sugerido:

> Sobre mim

O texto deve comunicar:

- interesse por desenvolvimento web;
- foco em resolver problemas;
- cuidado com qualidade e organização;
- comunicação clara;
- responsabilidade;
- respeito aos prazos;
- aprendizado contínuo;
- interesse em construir relações profissionais duradouras.

Não inventar:

- datas;
- empresas;
- cargos;
- cursos;
- resultados;
- tempo de experiência.

Texto-base adaptável:

> Sou Pedro, Desenvolvedor Full Stack com foco em interfaces modernas e desenvolvimento de APIs. Gosto de entender o problema antes de começar a desenvolver, buscando criar soluções organizadas, funcionais e fáceis de utilizar. Valorizo comunicação clara, responsabilidade, cumprimento de prazos e evolução constante.

Essa seção pode conter poucos destaques visuais, como:

```text
Comunicação clara
Compromisso com prazos
Código organizado
Aprendizado contínuo
```

Não utilizar números ou estatísticas sem dados reais.

---

## 13. Serviços

Título obrigatório:

> O que posso fazer por você?

Apresentar os seguintes serviços:

### Landing Pages

Páginas focadas em apresentar produtos, serviços ou campanhas com clareza e chamadas para ação.

### Sites Institucionais

Sites profissionais para apresentar empresas, negócios, serviços e formas de contato.

### Sistemas Web

Desenvolvimento de interfaces e funcionalidades para aplicações web de acordo com o escopo e as necessidades do projeto.

### Dashboards

Interfaces para visualizar indicadores, dados e informações de maneira clara e organizada.

### Painéis Administrativos

Áreas internas para gerenciamento de conteúdos, registros e operações.

### APIs REST

Desenvolvimento de APIs em Java ou JavaScript para operações, regras de negócio e acesso a dados.

### Regras dos cards

Cada serviço deve conter:

- ícone Lucide;
- título;
- descrição curta;
- benefício compreensível para o cliente.

Não utilizar descrições excessivamente técnicas.

Não prometer integração avançada entre Front-end e Back-end como diferencial principal.

---

## 14. Projetos

Esta deve ser a seção de maior destaque depois da Hero.

Título sugerido:

> Projetos selecionados

Descrição sugerida:

> Alguns projetos que representam minha forma de transformar necessidades em soluções digitais.

Exibir preferencialmente de três a seis projetos.

Cada projeto deve conter:

- nome;
- imagem ou captura real;
- contexto;
- problema ou objetivo;
- solução desenvolvida;
- contribuição de Pedro;
- tecnologias;
- link para demonstração, quando existir;
- link para repositório, quando puder ser público.

### Estrutura dos dados

```ts
type Project = {
  readonly id: string;
  readonly title: string;
  readonly summary: string;
  readonly context: string;
  readonly solution: string;
  readonly contribution: string;
  readonly technologies: readonly string[];
  readonly imageUrl: string;
  readonly imageAlt: string;
  readonly repositoryUrl?: string;
  readonly liveUrl?: string;
  readonly status?: ProjectStatus;
};
```

### Regras

- não inventar métricas;
- não inventar resultados;
- não exigir deploy para todos os projetos;
- não exigir repositório público;
- não exibir um botão quando o respectivo link não existir;
- não criar filtros com menos de seis projetos;
- não utilizar carrossel automaticamente;
- priorizar uma grade responsiva;
- tecnologias devem ser secundárias em relação ao problema e à solução;
- imagens devem possuir dimensões definidas;
- utilizar placeholders identificáveis enquanto as imagens reais não forem fornecidas.

Quando não houver projetos reais suficientes, utilizar projetos de estudo somente quando identificados claramente como:

> Projeto de estudo

Não apresentar projetos de estudo como trabalhos realizados para clientes.

---

## 15. Tecnologias

Título sugerido:

> Tecnologias e ferramentas

Organizar por contexto de uso, sem barras de progresso e sem porcentagens.

Categorias:

### Uso principal

Tecnologias utilizadas com maior frequência e segurança.

### Experiência prática

Tecnologias já utilizadas em projetos ou estudos práticos.

### Atualmente estudando

Tecnologias em processo de aprendizado.

Rust deve aparecer apenas em:

```text
Atualmente estudando
```

Não definir automaticamente quais tecnologias pertencem a cada grupo sem confirmação dos dados reais.

Possíveis categorias complementares:

```text
Front-end
Back-end
Banco de dados
Ferramentas
```

Evitar exibir uma nuvem extensa de ícones sem contexto.

---

## 16. Contato

Título sugerido:

> Vamos conversar sobre sua ideia?

Texto sugerido:

> Conte-me um pouco sobre o que você precisa. Podemos conversar sobre o projeto, entender os objetivos e avaliar a melhor forma de transformar a ideia em uma solução web.

A seção deve apresentar:

- CTA principal;
- e-mail;
- WhatsApp, quando fornecido;
- LinkedIn;
- GitHub;
- formulário de contato, quando configurado.

### Campos do formulário

```text
Nome
E-mail
Tipo de projeto
Mensagem
```

Opções possíveis de tipo de projeto:

```text
Landing Page
Site Institucional
Sistema Web
Dashboard
Painel Administrativo
API REST
Outro
```

### Funcionamento

Na primeira versão, não criar um Back-end próprio apenas para o formulário.

O envio poderá utilizar um serviço externo, desde que configurado com dados reais e sem expor credenciais.

Enquanto não houver um serviço configurado:

- manter contato por e-mail ou WhatsApp;
- não simular um envio bem-sucedido;
- não apresentar mensagens falsas de confirmação.

### Validação

- utilizar elementos nativos sempre que possível;
- exibir mensagens claras;
- associar erros aos respectivos campos;
- preservar os dados preenchidos quando houver erro;
- impedir envios duplicados;
- indicar estado de carregamento;
- informar sucesso ou falha;
- tornar todas as mensagens acessíveis a leitores de tela.

---

## 17. Footer

O Footer deve ser simples e conter:

- nome;
- título profissional;
- links sociais reais;
- navegação auxiliar;
- ano atual calculado dinamicamente;
- aviso de direitos autorais sem texto jurídico excessivo.

Exemplo:

> Desenvolvido por Pedro com React e TypeScript.

Não é necessário exibir todas as tecnologias utilizadas.

---

## 18. Direção visual

Utilizar tema escuro, minimalista e profissional.

A interface deve ter:

- boa quantidade de espaço negativo;
- forte hierarquia tipográfica;
- cards discretos;
- bordas suaves;
- sombras leves;
- uma única cor principal de destaque;
- alinhamentos consistentes;
- seções visualmente distintas sem excesso de divisores.

Evitar:

- neon;
- gradientes excessivos;
- glassmorphism intenso;
- partículas;
- fundos constantemente animados;
- elementos 3D;
- parallax;
- cursores personalizados;
- efeitos que dificultem a leitura;
- sombras muito fortes;
- excesso de cards;
- bordas em todos os elementos.


## 20. Tipografia

Utilizar:

- **Sora** para títulos e elementos de destaque;
- **Inter** para textos, botões, inputs e elementos de interface.

Fallbacks:

```css
--font-heading: "Sora", "Inter", system-ui, sans-serif;
--font-body: "Inter", system-ui, -apple-system, BlinkMacSystemFont, sans-serif;
```

Carregar apenas os pesos realmente utilizados.

Pesos recomendados:

```text
400
500
600
700
```

Evitar fontes muito finas em fundo escuro.

A tipografia deve ser fluida, utilizando `clamp()` quando apropriado.

Exemplo conceitual:

```css
font-size: clamp(2.5rem, 7vw, 5.5rem);
```

Não utilizar tamanhos arbitrariamente grandes em mobile.

## 21. Layout e responsividade

Adotar abordagem Mobile First.

Breakpoints compatíveis com Tailwind:

```text
sm
md
lg
xl
2xl
```

Requisitos:

- nenhum overflow horizontal;
- botões confortáveis para toque;
- cards reorganizados conforme a largura;
- textos sem linhas excessivamente longas;
- navegação adaptada para mobile;
- imagens responsivas;
- CTAs visíveis sem sobreposição;
- seções com padding proporcional;
- evitar alturas fixas;
- testar zoom de até 200%;
- não esconder conteúdo essencial em dispositivos menores.

O grid de doze colunas pode ser usado no desktop, mas não deve ser aplicado rigidamente em componentes simples.

---

## 22. Motion design

Utilizar animações sutis e funcionais.

Animações permitidas:

- fade-in;
- slide-up curto;
- transição de cor;
- alteração discreta de borda;
- leve mudança de escala em elementos interativos;
- revelação progressiva durante a rolagem.

Durações recomendadas:

```ts
const MOTION_DURATION_FAST = 150;
const MOTION_DURATION_DEFAULT = 250;
const MOTION_DURATION_SLOW = 400;
```

Easing recomendado:

```text
ease-out
```

Regras:

- animações não devem bloquear interação;
- não animar grandes áreas sem necessidade;
- evitar animações acima de 500 ms;
- não animar todos os elementos simultaneamente;
- utilizar atraso escalonado com moderação;
- respeitar `prefers-reduced-motion`;
- desativar movimentos não essenciais quando solicitado pelo sistema.

---

## 23. Acessibilidade

Buscar conformidade com WCAG 2.2 nível AA.

Requisitos:

- HTML semântico;
- apenas um `h1`;
- hierarquia correta de headings;
- navegação completa por teclado;
- foco visível;
- ordem de tabulação lógica;
- link para pular ao conteúdo;
- labels explícitos;
- mensagens de erro associadas aos campos;
- textos alternativos significativos;
- áreas clicáveis adequadas;
- contraste suficiente;
- nenhuma informação transmitida apenas por cor;
- suporte a `prefers-reduced-motion`;
- não utilizar `div` clicável quando um `button` ou `a` for apropriado;
- não adicionar ARIA quando elementos nativos já resolverem;
- utilizar `aria-label` somente quando o nome acessível não estiver presente visualmente;
- modais e menus devem controlar corretamente o foco;
- links externos devem ser identificáveis pelo contexto.

---

## 24. Arquitetura Front-end

Utilizar **Feature-Based Architecture com componentes compartilhados**.

Atomic Design pode ser usado apenas como referência conceitual. Não criar obrigatoriamente diretórios `atoms`, `molecules` e `organisms`.

Não aplicar Clean Architecture completa em um portfólio essencialmente visual.

Estrutura recomendada:

```text
src/
├── app/
│   ├── App.tsx
│   └── main.tsx
├── assets/
│   ├── images/
│   └── icons/
├── components/
│   ├── layout/
│   └── ui/
├── features/
│   ├── about/
│   ├── contact/
│   ├── hero/
│   ├── projects/
│   ├── services/
│   └── technologies/
├── hooks/
├── lib/
├── constants/
├── styles/
├── types/
└── utils/
```

Não criar diretórios vazios.

Não criar um arquivo separado quando o conteúdo for utilizado apenas uma vez e permanecer pequeno.

---

## 25. Design patterns permitidos

Aplicar somente quando fizer sentido:

- Component Composition;
- Feature-Based Architecture;
- Single Responsibility;
- Separation of Concerns;
- Custom Hooks;
- Data-Driven Components;
- Controlled Components para formulários;
- Dependency Inversion somente quando existir dependência relevante;
- Early Return;
- composição em vez de herança.

Não aplicar obrigatoriamente:

- Repository Pattern;
- Factory Pattern;
- Clean Architecture completa;
- Container/Presentational para cada componente;
- Atomic Design rígido;
- abstrações genéricas para componentes utilizados uma única vez.

Regra:

> Não criar uma abstração até existir uma necessidade concreta, repetição relevante ou ganho claro de legibilidade.

## 26. TypeScript

Ativar modo estrito.

Configuração esperada:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true
  }
}
```

Regras:

- evitar `any`;
- preferir `unknown` quando o tipo realmente for desconhecido;
- validar dados antes de utilizá-los;
- utilizar `readonly` em dados imutáveis;
- evitar assertions desnecessárias;
- não utilizar `as` apenas para silenciar erros;
- inferir tipos simples;
- tipar explicitamente contratos e dados de domínio;
- utilizar unions para estados conhecidos;
- não criar enums quando uma união literal for mais simples.

Exemplo:

```ts
type ProjectStatus = "completed" | "in-progress" | "study";
```

## 27. Convenções de nomenclatura

### Constantes

Todas as constantes devem seguir obrigatoriamente:

```text
SCREAMING_SNAKE_CASE
```

Exemplos:

```ts
const MAX_PROJECTS = 6;
const DEFAULT_THEME = "dark";
const MOTION_DURATION_DEFAULT = 250;
const SITE_NAME = "Pedro | Desenvolvedor Full Stack";
```

Essa regra se aplica a:

- constantes locais;
- constantes exportadas;
- objetos de configuração imutáveis;
- arrays estáticos;
- valores de ambiente normalizados;
- mapas de configuração.

Exemplos:

```ts
const NAVIGATION_ITEMS = [];
const PROJECT_STATUS_LABELS = {};
const CONTACT_FORM_DEFAULT_VALUES = {};
```

Não extrair automaticamente para constante:

- valores triviais utilizados apenas uma vez;
- strings locais sem significado de domínio;
- propriedades JSX simples;
- valores cuja extração reduza a legibilidade.

### Variáveis

Utilizar `camelCase`:

```ts
const selectedProject = projects[0];
const isMenuOpen = false;
```

### Funções

Utilizar `camelCase`:

```ts
function handleContactSubmit() {}
function scrollToProjects() {}
```

### Componentes

Utilizar `PascalCase`:

```text
HeroSection
ProjectCard
ContactForm
```

### Hooks

Iniciar obrigatoriamente com `use`:

```text
useTypingEffect
useReducedMotion
useIntersectionObserver
```

### Types

Utilizar `PascalCase` sem prefixo `I`:

```ts
type Project = {};
type ContactFormData = {};
```

Não utilizar:

```text
IProject
IContactFormData
```

### Pastas

Utilizar `kebab-case`:

```text
project-card/
contact-form/
hero-section/
```

### Arquivos

Componentes:

```text
ProjectCard.tsx
HeroSection.tsx
```

Hooks:

```text
useReducedMotion.ts
useTypingEffect.ts
```

Utilitários:

```text
formatProjectUrl.ts
validateContactForm.ts
```

Constantes:

```text
navigation.constants.ts
projects.constants.ts
```

Tipos:

```text
project.types.ts
contact.types.ts
```

## 28. Padrões de código

Obrigatório:

- código legível;
- componentes funcionais;
- TypeScript estrito;
- early returns quando melhorarem a clareza;
- funções pequenas e focadas;
- responsabilidades bem definidas;
- dados estáticos separados da apresentação quando isso melhorar a manutenção;
- tratamento explícito de erros;
- nomes descritivos;
- reutilização baseada em necessidade real.

Evitar:

- `any`;
- `console.log` no código final;
- código morto;
- imports não utilizados;
- strings duplicadas relevantes;
- números mágicos com significado de domínio;
- funções extensas;
- componentes excessivamente grandes;
- comentários explicando código óbvio;
- props booleanas ambíguas;
- condicionais profundamente aninhadas;
- uso desnecessário de `useEffect`;
- armazenamento em estado de valores que podem ser derivados;
- `dangerouslySetInnerHTML`.

## 29. Organização interna dos componentes

Utilizar a seguinte ordem quando aplicável:

```text
1. Imports
2. Types
3. Constants
4. Component
5. Hooks
6. State
7. Refs
8. Derived values
9. Effects
10. Handlers
11. Return JSX
```

Não adicionar comentários separadores em componentes pequenos.

Hooks devem seguir as regras oficiais do React:

- sempre no nível superior;
- nunca dentro de condicionais;
- nunca depois de retornos condicionais;
- nunca dentro de loops.

## 30. Imports

Organizar os imports em grupos:

```text
1. React e bibliotecas externas
2. Aliases internos
3. Componentes
4. Hooks
5. Utilitários e constantes
6. Types
7. Estilos e assets
```

Separar grupos com uma linha vazia quando isso melhorar a leitura.

Utilizar aliases como:

```text
@/components
@/features
@/hooks
@/types
@/constants
```

Evitar caminhos relativos extensos:

```ts
import { Button } from "../../../../components/ui/Button";
```

## 31. Conteúdo orientado por dados

Serviços, projetos, tecnologias e navegação devem ser definidos como dados quando houver repetição estrutural.

Exemplo:

```ts
const SERVICES = [
  {
    id: "landing-pages",
    title: "Landing Pages",
    description: "Páginas focadas em apresentar uma oferta com clareza.",
    icon: LayoutTemplate,
  },
] as const;
```

Como `SERVICES` é uma constante, deve seguir `SCREAMING_SNAKE_CASE`.

Não armazenar componentes React serializados em arquivos JSON.

## 32. Estado e efeitos

Regras:

- não utilizar estado para valores calculáveis;
- não utilizar `useEffect` para sincronizações que podem ocorrer em handlers;
- limpar timers, observers e listeners;
- evitar múltiplos estados booleanos para representar o mesmo fluxo;
- preferir unions para estados exclusivos.

Exemplo:

```ts
type SubmissionStatus = "idle" | "submitting" | "success" | "error";
```

Não utilizar:

```ts
const [isLoading, setIsLoading] = useState(false);
const [isSuccess, setIsSuccess] = useState(false);
const [isError, setIsError] = useState(false);
```

quando apenas um estado pode existir por vez.

- não dividir uma aplicação pequena em chunks excessivos;
- otimizar ícones e assets;
- evitar efeitos contínuos durante a rolagem;
- respeitar o orçamento de performance.
